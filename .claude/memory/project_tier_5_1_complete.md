# Tier 5.1 — a3-unpublish-deliberate — shipped 2026-06-08

Roadmap: [[../../docs/superpowers/specs/2026-06-07-polish-and-bugfix-roadmap.md]] Tier 5.1. Source: B.4 follow-up #1 in [[project-b4-complete]].

## What shipped

New interactive command `a3-unpublish-deliberate (file-or-id)` in `a3madkour-publish-deliberate.el`. Synchronous (no async-publish UI / spinner) — recovery wants immediate feedback. Composes existing primitives:

1. **Resolve to id** via `a3madkour-pub-deliberate--resolve-to-id`:
   - nil / non-string / empty string → nil → `user-error`
   - existing file → look up id via `a3madkour-pub/note-metadata`
   - anything else → treated as opaque id (manifest contract doesn't require UUID format; recovery often runs after the source file is gone)
2. **Look up manifest entry** via `a3madkour-pub-history--find-note-by-id`.
3. **Validate**: entry exists, state ∈ {live, draft}, URL parses to (section . slug), section is in `a3madkour-pub-deliberate--handlers`.
4. **Delete bundle** via `a3madkour-pub--unpublish-delete-bundle`; mirrors bug-1.1's contract: `'failed` → `user-error` + manifest stays at prior state.
5. **Record removed** via `a3madkour-pub-history/record-publish id nil 'removed`.

Bundle-already-absent (stale-manifest recovery) is NOT a failure — manifest still advances to `removed` so state converges.

## Why this design

- **Synchronous, not async**: recovery vs routine publish; no need for `*a3-publish*` buffer / spinner. The async lifecycle wrapper would also drag in `begin-publish` / `finish-publish` semantics that don't fit a one-id recovery.
- **Living-section guard**: garden / library / research notes belong to publish-living's diff. Unpublishing them via this command would race the next publish-living run. Authors instead unmark `#+HUGO_PUBLISH:` in the source and re-run publish-living.
- **No `data/*.yaml` refresh**: this command only touches the Hugo content bundle + the manifest. citations.yaml and per-medium aggregates are regenerated by publish-living; author re-runs that if they need data files refreshed. Documented in the docstring.
