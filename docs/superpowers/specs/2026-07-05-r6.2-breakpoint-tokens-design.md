# R6.2 — Breakpoint tokens (design)

**Date:** 2026-07-05
**Roadmap row:** R6.2 (`docs/superpowers/specs/2026-07-03-audit-remediation-roadmap.md`, Tier R6, Severity Low, optional design-scale polish).
**Status:** designed.

## Problem

Desktop breakpoints in `assets/css/main.css` are one-off values with no shared
vocabulary, and the JS side mirrors two of them as magic numbers that hand-sync
with CSS by luck:

**CSS `@media` widths (current):**
- Mobile (disciplined, leave as-is): `max-width` 720 (×13), 600 (×4), 480 (×4).
- Desktop one-offs (each ×1, per-component): `max-width` 800 / 900 / 960 / 1099
  / 1140 / 1219; `min-width` 960 / 1100 / 1280.

Each desktop breakpoint drives ONE element's layout change:

| Width | Element |
|---|---|
| `max 800` | `.home-hero` reflow |
| `max 900` | `.essay-grid` bento reflow |
| `max 960` / `min 960` | `.library-cat-cards` / `.streams-archive-list` |
| `max 1099` + `min 1100` | `.essay-body .sidenote` margin ↔ inline toggle |
| `max 1140` | `.essay-figure.wide` constrain |
| `max 1219` | `.page-sidebar` rail hide |
| `min 1280` | `.streams-archive-list` wide |

**JS breakpoint constants / literals (hand-synced with CSS):**
- `assets/js/essay.js`: `RAIL_BREAKPOINT = 1100` (sidenote/rail; mirrors the CSS
  1100 sidenote toggle).
- `assets/js/citation-card.js`: `MOBILE_BREAKPOINT = 720`.
- `matchMedia('(max-width: 720px)')` literals in `garden-stack.js`,
  `graph-core.js`, `works.js`, `garden-pathlog-popover.js`.

Nothing prevents a future `@media (max-width: 905px)` one-off, or the JS `1100`
silently drifting from the CSS `1100`. That silent CSS↔JS divergence is the real
risk.

## Constraint that shapes the whole approach

**CSS `@media` queries cannot use `var()`** — `@media (max-width: var(--bp-md))`
is invalid, and there is no Sass/PostCSS in this hand-rolled pipeline. So there
is no runtime-token mechanism for the CSS side. Breakpoint "tokens" can only be
a *documented scale* enforced by a *linter*, not runtime custom properties.

## Goal (Option B1 — document + guard, hold layout)

Establish a canonical breakpoint scale, document it, and add a guard linter that
fails CI on any breakpoint outside it. **Zero layout change and zero runtime
change**: the only edits to shipped files are inert comments; the only real code
is the non-shipping linter.

Rejected alternatives:
- **Option A (consolidate + snap the one-offs to tiers):** these are
  per-component content breakpoints, so moving them shifts where each layout
  change fires — real per-component visual risk for little gain on a site that
  renders fine. Declined.
- **Option B2 (expose `--bp-*` custom props + rewire JS to read them via
  `getComputedStyle`):** touches shipped JS runtime and adds a parse, yet the
  props still can't reach the `@media` queries (the CSS limitation), so the
  linter must cross-check literals anyway. More machinery for the same
  guarantee. Declined in favour of B1's linter-only enforcement.

## Canonical breakpoint scale

The documented vocabulary new code should use:

| Tier | Width | Role |
|---|---|---|
| `xs` | 480px | mobile small |
| `sm` | 600px | mobile |
| `md` | 720px | mobile/tablet boundary (the site's primary responsive line; JS mobile) |
| `lg` | 960px | narrow desktop / half-screen (the tiling-WM tier the author tests at) |
| `xl` | 1100px | rail + sidenote margin toggle (JS `RAIL_BREAKPOINT`) |
| `2xl` | 1280px | wide desktop |

## Allowlisted per-component exceptions (CSS-only, kept as-is)

Deliberate deviations, each documented with its reason (mirrors R6.1's
micro-nudge allowlist). These stay literal; the linter accepts them but new
additions require a justified allowlist entry:

- `800` — `.home-hero` reflow
- `900` — `.essay-grid` bento reflow
- `1140` — `.essay-figure.wide` constrain
- `1219` — `.page-sidebar` rail hide

**Seam rule (general, not an allowlist entry):** a `max-width: V` is valid when
`V + 1 ∈ canonical` — the standard `max = min − 1` non-overlap pattern. This
accepts `1099` as the partner of `min-width: 1100` without a bespoke entry, and
generalises to any future canonical seam.

## Guard linter — `tools/check_breakpoints.py` (32nd pair)

`check_breakpoints.py` + `test_check_breakpoints.py`, following the R5.4
`run(repo_root) -> (int, list[str])` seam + thin `main()`. Stdlib only.

Module-level constants (single source of truth for enforcement):
- `CANONICAL: set[int] = {480, 600, 720, 960, 1100, 1280}`
- `CSS_ALLOWLIST: dict[int, str] = {800: "...", 900: "...", 1140: "...", 1219: "..."}`

**CSS check** (`assets/css/main.css`):
- Extract every width feature from `@media` preludes:
  `(min-width|max-width)\s*:\s*(\d+)px`.
- A value `N` is valid iff `N ∈ CANONICAL`, or `N ∈ CSS_ALLOWLIST`, or
  (for a `max-width`) `N + 1 ∈ CANONICAL` (seam rule).
- Any other value is a violation.

**JS check** (`assets/js/*.js`, excluding `vendor/`):
- Parse `*_BREAKPOINT` constant assignments (`const NAME_BREAKPOINT = <int>`)
  and every `matchMedia('...(min-width|max-width: <int>px)...')` width literal.
- Each parsed value must be in `CANONICAL`. (JS must not reference the CSS-only
  per-component exceptions — those are layout-tweak breakpoints with no JS
  behaviour attached. A future JS `matchMedia('(max-width: 900px)')` or a
  `RAIL_BREAKPOINT` drift to `1099` is therefore a violation.)
- This is the anti-drift teeth: CSS↔JS divergence becomes a CI failure.

**Precision:** the JS scan targets only `*_BREAKPOINT` constants and `matchMedia`
width literals, so unrelated numbers (`VIEWPORT_PAD`, `innerWidth * 0.8`, timing
constants) are never considered.

Returns `(1, [errors])` on any violation, `(0, [])` when clean. The current tree
passes (all JS breakpoints are 720/1100, both canonical; all CSS widths are
canonical, allowlisted, or the 1099 seam partner).

## Documentation (the "document" half)

- **`main.css`:** a breakpoint-scale comment header (near the top / tokens
  section) listing the six canonical tiers, the four per-component exceptions
  with reasons, the seam rule, and a note that `tools/check_breakpoints.py`
  enforces the scale and the CSS↔JS contract.
- **`essay.js` / `citation-card.js`:** a one-line cross-ref comment on
  `RAIL_BREAKPOINT` / `MOBILE_BREAKPOINT` pointing at the linter and the CSS
  tier it mirrors.

No other shipped-file edits. No `@media` value moves. No JS runtime change.

## Wiring

  spacing-token pair (R6.1), matching the sibling two-step pattern.
- `tools/ci-local.sh`: the two-line pair after the spacing-token pair.
- `CLAUDE.md`: linter count "Thirty-one" → "Thirty-two"; add `breakpoints` to

## Verification

R6.2 is additive — no layout or runtime change — so the guard is simpler than
R6.1 (no visual spot-check needed):

1. `python3 tools/check_breakpoints.py` → exits 0 on the current tree.
2. `python3 -m unittest tools/test_check_breakpoints.py` → all pass (fixtures
   exercise: canonical ok, allowlist ok, seam ok, a bad CSS value flagged, a
   bad JS `matchMedia`/`*_BREAKPOINT` value flagged, `prefers-*` media queries
   with no width ignored, `innerWidth`/timing numbers ignored).
3. `tools/ci-local.sh` green — full linter suite (incl. the new pair) + the
   8-spec Playwright E2E **unchanged at 9/9** (proves no runtime behaviour moved,
   since only comments were added to shipped files) + build.
4. `python3 tools/check-contrast.py` green (unaffected; part of the gate).

## Out of scope / deferred

- Moving/consolidating any `@media` value (that is Option A — declined).
- `--bp-*` custom properties / JS `getComputedStyle` rewiring (Option B2 —
  declined).
- R6.3 built-HTML link crawl (trigger-gated, separate).
- Mobile breakpoints (480/600/720) — already disciplined; only documented, not
  changed.
