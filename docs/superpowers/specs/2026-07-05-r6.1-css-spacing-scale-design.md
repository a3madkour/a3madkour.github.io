# R6.1 — CSS spacing scale (design)

**Date:** 2026-07-05
**Roadmap row:** R6.1 (`docs/superpowers/specs/2026-07-03-audit-remediation-roadmap.md`, Tier R6, Severity Low, optional design-scale polish).
**Status:** designed.

## Problem

`assets/css/main.css` (5636 lines, §1–§49) has **no spacing tokens**. Raw rem
values are scattered across `gap` / `padding` / `margin`:

- ~40 distinct rem values across the three properties.
- A long tail of near-duplicates doing the same job in different places
  (0.35 / 0.4 / 0.45; 0.7 / 0.75 / 0.8 / 0.85 / 0.9; 1.2 / 1.25; 1.4 / 1.5)
  — the "asymmetric rhythm" the audit flagged.
- px-conversion artifacts (0.625, 0.875, 0.125) and sub-pixel micro-nudges
  (0.02, 0.05, 0.1, 0.15rem).

There is no single place to tune spacing density and no discipline preventing
new magic numbers from accreting.

## Goal (Option A — rationalized scale)

Introduce a named `--space-*` scale and migrate spacing declarations to it,
**snapping** the rare tail of near-duplicates to their nearest step. This is a
deliberate visual change (sub-pixel to ~4px shifts on rarely-used values), not a
pure refactor — the payoff is a real density knob, a rationalized rhythm, and a
guard against future drift.

Rejected alternative (Option B, alias-only): tokens that exactly equal current
values deliver only indirection — no density knob, no rhythm fix. Not worth the
churn.

## The token ladder

Twelve t-shirt steps, defined **once on `:root`** in the tokens section.
Spacing is mode-independent, so — unlike the color tokens — these are **not**
duplicated in the `:root[data-theme="dark"]` block or the
`@media (prefers-color-scheme: dark)` block.

| Token | Value | px | Absorbs (snapped from) |
|---|---|---|---|
| `--space-3xs` | 0.25rem | 4 | 0.2, 0.3 |
| `--space-2xs` | 0.4rem | 6.4 | 0.35, 0.45 |
| `--space-xs` | 0.5rem | 8 | 0.55 |
| `--space-sm` | 0.6rem | ~10 | 0.625 |
| `--space-md` | 0.75rem | 12 | 0.7, 0.8, 0.85, 0.875, 0.9 |
| `--space-lg` | 1rem | 16 | 1.1, 1.125 |
| `--space-xl` | 1.25rem | 20 | 1.2 |
| `--space-2xl` | 1.5rem | 24 | 1.4, 1.6 |
| `--space-3xl` | 2rem | 32 | 1.75, 1.8 |
| `--space-4xl` | 2.5rem | 40 | — |
| `--space-5xl` | 3rem | 48 | — |
| `--space-6xl` | 4rem | 64 | — |

The high-frequency values (0.4, 0.5, 0.6, 0.75, 1, 1.25, 1.5, 2, 2.5, 3, 4)
are named steps and **do not move**. Only the low-frequency tail snaps;
worst-case snap is ~2–4px on values that appear once or twice (0.9→0.75,
1.75→2). The ladder is a **non-uniform literal scale** (0.4 and 0.6 are not
0.25-multiples), so the "density knob" is the twelve token definitions in one
place — not a single calc multiplier. That is an accepted, documented trade-off:
preserving the author's established rhythm matters more than a pure geometric
scale.

## Migration scope

**In scope** — rem-valued declarations of:

- `gap`, `row-gap`, `column-gap`
- `padding` and per-side (`padding-top` etc.)
- `margin` and per-side (`margin-top` etc.)

Multi-value shorthands map each component:
`padding: 0.5rem 1rem` → `padding: var(--space-xs) var(--space-lg)`.

**Out of scope** (left as literals):

- Sub-0.25rem micro-nudges (0.02 / 0.05 / 0.1 / 0.125 / 0.15rem) — these are
  hairline border/alignment tweaks, not layout rhythm; snapping them would break
  things. They are the guard-linter allowlist.
- Non-rem spacing (px / em / ch) — the scale is rem-based.
- Positioning insets (`top` / `left` / `right` / `bottom` / `inset`) — position,
  not rhythm.
- rem literals inside non-trivial `calc()` — migrated only where the mapping is
  obvious and safe; left alone if gnarly (noted case-by-case during impl).

## Execution

All three properties migrated in the one R6.1 slice, **batched by CSS section
(§1–§49)** for reviewability. Done **inline** (mechanical but parity-sensitive),
not subagent fan-out: one careful pass per batch → dev-server eyeball → next
batch. The `--space-*` block is added to `:root` first; batches follow.

## Guard linter (31st linter pair)

`tools/check_spacing_tokens.py` + `tools/test_check_spacing_tokens.py`, following
the established `run(repo_root) -> (int, list[str])` seam + thin `main()`
(R5.4 convention).

**Invariant:** no raw rem value appears in a `gap` / `row-gap` / `column-gap` /
`padding*` / `margin*` declaration in `main.css` except:

1. values that exactly equal a `--space-*` token (these should already be
   tokenized — flagged, since a literal that matches a token is drift), and
2. the sub-0.25rem micro-nudge allowlist (0.02, 0.05, 0.1, 0.125, 0.15rem).

Effectively: every in-scope rem spacing value must be either a `var(--space-*)`
reference or an allowlisted micro-nudge. Any bare rem ≥ 0.2rem in an in-scope
property fails the linter. This prevents future magic-number accretion.

Wire it into CI (pre-build linters block) and `tools/ci-local.sh` alongside the
other pairs.

## Verification

R6.1 changes pixels, so the byte-identical minified-diff guard used for pure
template refactors does **not** apply. The guard is three-legged:

1. **`tools/ci-local.sh`** green — includes the new spacing linter pair, the
   full linter suite, and the 8-spec Playwright E2E (functional regression:
   confirms no `var(--space-*)` typo or collapsed layout breaks a graph mount,
   chip strip, search listbox, or cite modal). E2E asserts behavior, not pixels
   — it is a regression guard, not the visual verifier.
2. **`tools/check-contrast.py`** green — spacing doesn't touch color, but it
   stays in the gate.
3. **Dev-server spot-checks** — the actual visual verifier, author-driven, with
   a per-batch "what to eyeball" checklist, tested at ~960px half-screen
   (author runs a tiling WM) as well as full width. This is where the snapped
   values get judged.

No screenshot-baseline visual regression in this slice: Option A intentionally
moves pixels, so baselines would fight the migration. Automated visual snapshots
are a defensible follow-up once the look is settled.
