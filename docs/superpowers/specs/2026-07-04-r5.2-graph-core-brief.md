## Proposed boundary

A shared `assets/js/graph-core.js` exporting a factory `createGraph(options)` → `{ rebuild, resetView, resetPositions, openPanel, closePanel }`, plus three thin per-section adapters that keep only their divergent surface.

**Cleanly shared (extract as-is, no parameterization beyond class names):**
- Byte-identical across all 3: `persistCacheDebounced`, `flushCache`, `reducedMotion`, `isMobile`, `rebuildGraph`, `resetView`, `resetPositions`.
- Structurally identical (rename only): `getActiveCanvas`, `loadCachedPositions`, `saveCachedPositions`, `applySavedPanelWidth`.
- Inside `buildSimulation` (the big win): the d3 dynamic-import block, the d3-selection `interrupt` stub, canvas-dimension read, the full 44-line `dragBehavior`, the 20-line `zoomBehavior`, cache-hit restore, `renderTick`, and the cache-miss settle loop (`for i<300 sim.tick()`) are all identical.

**Stays in adapters (the divergent surface):**
- `parseData` (each consumes a different node/edge JSON shape — see below).
- `applyFilters` + `state.filters` shape (garden: tag/stage/local; research: tag[]/status; works: medium).
- Node-shape rendering in `buildSimulation` (garden: circle+`tagColor` fill; research: rect/circle by kind + `data-theme-color`; works: rounded-rect + `<use href="#g-{medium}">` glyph + `<defs>` gradient).
- Node color assignment, click destination (garden: dispatch stack event / research: build URL by kind / works: `node.url`), and `nodeRadius` (works uses `badgeSize`, not degree).

**Garden-only, gate behind options/hooks:** stack-coordination (`openSlugForElement` custom-event path, `updateInStackMarkers`, `garden:stack-changed` listener, `state.inStack`), N-hop local mode (`bfsNeighborhood`, `state.filters.local`, note-page scope chip), dynamic JS legend (`buildLegend`). Research/works pass no-ops.

## Unified data contract

Adapters normalize to one node type before handing to core; core stays data-shape-agnostic. Superset of optional fields: `slug, title, url?, degree?, tag?, tags?, kind?, stage?, status?, medium?, featured?, themeColorIdx?`. Edge superset: `source, target, crossTopic?|kind?, via?, weight?, shared?`. Core only needs `{slug, x, y}` for layout + whatever the adapter's renderer reads.

## Four open architecture decisions (resolve in brainstorm)

**Works is the outlier on the first three** — garden+research already agree. The cleanest path may be to *normalize works to the garden/research convention first, then extract* — but that's a decision, not a given.

1. **Toolbar: build vs wire.** Garden/research JS-build chips (`makeFilterChip`/`buildFilterChips`); works pre-renders chips in `works/graph-panel.html` and only wires handlers (`wireToolbar`). Options: (a) standardize on JS-build (rewrite works panel to empty toolbar), (b) standardize on SSR-wire (pre-render garden/research chips), (c) support both via `toolbar: 'build'|'wire'` option.
2. **Panel closed-state.** Garden/research use `aria-hidden + inert`; works uses the `hidden` attribute. `inert` is more semantically correct — standardize on it, or keep an option?
3. **Canvas element.** Garden/research pass a `<div>` wrapper (core creates the `<svg>` inside); works passes the `<svg>` itself as the canvas. Standardize on the div-wrapper, or branch on `canvasIsSvg`?
4. **Test coverage before refactor (Risk 8 — recommend YES).** `graph.spec.ts` currently asserts only the **garden** mount. A broken research/works adapter would pass CI silently. Strongly recommend adding `/research/graph/` and `/works/graph/` mount specs (generic, like the garden one) *as the first step of R5.2* so the extraction has real coverage on all three. This is cheap and directly enabled by the R5.1 harness.

## Hard constraints (from recon, non-negotiable)

- **`graph-core.js` must live at `assets/js/` depth** — the runtime `await import('./vendor/d3-*.min.js')` path is relative to the compiled bundle at `js/`. A subdirectory (`js/core/`) breaks the d3 import. (Risk 6.)
- The d3-selection `interrupt` stub moves into core (done once; all adapters share via module cache). (Risk 7.)
- Core must accept a **class-name map / `classPrefix`** — per-section CSS (§28/§31/§36) keys off `garden-graph-node` / `research-graph-node` / `works-graph-node` etc.; core must emit the right classes. The `zoomBehavior.filter`'s `.closest('.{prefix}-graph-node')` is parameterized too. (Risk 1.)
- No cross-section state collision risk: each section is a separate esbuild bundle; only one graph loads per page. (Risk 9.)

## Definition of done

`graph-core.js` + 3 adapters replace the 3 monolith files with no behavioral change; `tests/e2e/graph.spec.ts` (garden) stays green and new research/works mount specs pass; LOC drops materially (~2280 → est. ~900–1100). Then R5.3 (AMS-block + graph-panel consolidation) opens.
