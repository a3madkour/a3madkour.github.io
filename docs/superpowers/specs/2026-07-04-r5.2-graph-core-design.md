# R5.2 ŌĆö graph-core extraction (design)

**Date:** 2026-07-04
**Status:** Design approved; ready for `superpowers:writing-plans`.
**Parent:** row R5.2 in [`2026-07-03-audit-remediation-roadmap.md`](2026-07-03-audit-remediation-roadmap.md). Supersedes the scoping brief [`2026-07-04-r5.2-graph-core-brief.md`](2026-07-04-r5.2-graph-core-brief.md) (recon + open decisions ŌĆö all now resolved).

## Goal

Collapse three near-duplicate d3 force-directed graph runtimes (`garden-graph.js` 793, `research-graph.js` 761, `works-graph.js` 726 ŌĆö **2,280 LOC, ~70ŌĆō75% shared**) into one shared `graph-core.js` + three thin adapters, with **no behavioral change**. Target ~900ŌĆō1,100 LOC. Guarded by the R5.1 E2E harness (mount specs now exist for all three graphs).

## Key decision (resolved in brainstorm)

**Normalize works to the garden/research conventions** rather than parameterize the core for both. Works's three divergences (SSR-wired toolbar, `hidden`-attribute panel state, svg-as-canvas) are copy-drift, not intent ŌĆö so the core gets **one code path per concern** and works's templates/CSS move to the shared convention. Leanest core, best dedup.

## Architecture

### Module boundary

One new `assets/js/graph-core.js`. **Hard constraint:** it must live at `assets/js/` depth ŌĆö the runtime `await import('./vendor/d3-*.min.js')` path is relative to the compiled bundle, which sits at `js/`. A subdirectory would break the d3 import.

It exports a factory `createGraph(adapter)` ŌåÆ controller `{ rebuild, resetView, resetPositions, openPanel, closePanel }`. The three existing files shrink to thin adapters that build an `adapter` object and call the factory.

### Core owns (extract verbatim ŌĆö identical across all three today)

- d3 dynamic-import block + the d3-selection `interrupt` stub (done once; shared via module cache).
- Position-cache machinery: `persistCacheDebounced`, `flushCache`, `loadCachedPositions`, `saveCachedPositions`, cache-key plumbing.
- `reducedMotion`, `isMobile`, `getActiveCanvas` (standalone page vs panel).
- The 44-line drag behavior (incl. Shift-pin, `persistCacheDebounced`), the 20-line zoom behavior, the cache-hit restore block, `renderTick`, and the cache-miss settle loop (`for i<300 sim.tick()`).
- SVG scaffold (edge/node layers), `resetView`, `resetPositions`, `rebuild`.
- Panel open/close ŌĆö **standardized on `aria-hidden + inert`** ŌĆö and panel-resize (`applySavedPanelWidth` + drag handle).
- Shared chip primitives: `makeFilterChip`, `makeActionChip`, `buildActionChips` (Reset view / Reset positions + hint).

Core parameterizes only via **`classPrefix`** (`"garden"|"research"|"works"`): it emits `${prefix}-graph-node`, `${prefix}-graph-edge`, `${prefix}-graph-nodes/edges` layers, and the zoom filter keys off `.${prefix}-graph-node`. This preserves the existing CSS ┬¦28/┬¦31/┬¦36 selectors byte-for-byte.

### Adapters own (the divergent ~175ŌĆō225 lines each)

The `adapter` object supplies:
- `classPrefix: string`
- `parseData(): {nodes, edges}` ŌĆö reads the section's JSON `<script>` blob and normalizes to the unified node/edge shape.
- `applyFilters(state): {nodes, edges}` ŌĆö the section's filter dimensions (garden: multi-tag + stage + local; research: tag[] + status; works: medium).
- `filterCacheKey(state): string` ŌĆö the filter fragment folded into the position-cache key.
- `nodeRadius(node): number` ŌĆö garden/research from degree; works from badge size.
- `renderNode(g, node): void` ŌĆö draws the node's shape/color/glyph into the core-created `<g>` (garden: circle + `tagColor` fill; research: rect/circle by kind + `data-theme-color`; works: rounded-rect + `<use href="#g-{medium}">` glyph).
- `forceParams: { linkDistance, linkStrength, chargeStrength, collideRadius(node) }`.
- `onNodeClick(node, el): void` ŌĆö navigation (required hook).
- **Optional lifecycle hooks** (core calls if present, else no-op): `onSvgCreate(svg)`, `onRenderComplete(controller)`, `onOpenPanel(root)`.

### Unified node/edge shape

Adapters normalize to a superset before handing to core; core only needs `{slug, x, y}` for layout plus whatever the adapter's `renderNode` reads. Optional node fields: `slug, title, url?, degree?, tag?, tags?, kind?, stage?, status?, medium?, featured?, themeColorIdx?`. Optional edge fields: `source, target, crossTopic?|kind?, via?, weight?, shared?`.

### Garden-only features (via adapter + hooks, not baked into core)

- **Stack-coordination:** garden's `onNodeClick` dispatches `garden:graph-navigate` when a `.garden-stack` exists (others `location.assign`); an `onRenderComplete` hook re-syncs `.in-stack` on nodes; garden's adapter-init wires the `garden:stack-changed` listener + `state.inStack`/`lastPointerInStack`. Core never learns about stacks.
- **N-hop local mode:** entirely inside garden's `applyFilters` (BFS) + a "Scope" chip it declares + its `filterCacheKey` fragment. To the core it is just another filter dimension.
- **Dynamic JS legend:** garden's `onOpenPanel(root)` fills `.graph-legend-colorkey[data-graph-legend-dynamic]` with swatches. Research/works legends stay SSR (no hook).

## Works normalization ŌĆö concrete template/CSS changes

1. **Toolbar:** `layouts/partials/works/graph-panel.html` emits an empty toolbar div (like garden/research); the works adapter JS-builds a single-select "medium" chip row + action chips via core primitives; delete `wireToolbar` + the pre-rendered chip/resize markup.
2. **Panel state:** works panel switches from the `hidden` attribute to `aria-hidden + inert`; drop `[hidden]`-based show/hide rules in ┬¦36/┬¦44 so it uses the shared inert pattern.
3. **Canvas:** `works/graph.html` (+ panel) changes `<svg class="works-graph-canvas">` ŌåÆ `<div class="works-graph-canvas">`; core creates the `<svg>` inside; works's gradient `<defs>` moves into the `onSvgCreate(svg)` hook; the `canvas.tagName === 'svg'` special-case is deleted.

**Test consequence:** `tests/e2e/works-graph.spec.ts` (currently asserts `svg.works-graph-canvas`) updates to the div-wrapper form (`.works-graph-canvas` containing `svg`) **as part of the works task** ŌĆö it still guards the real invariant (nodes mount); only the "where the svg lives" selector changes, because works's DOM intentionally changes. Garden/research specs are unaffected.

## Migration & verification

**Staged order** (each stage independently committable + testable):
1. Extract `graph-core.js` against garden (richest superset) + migrate the garden adapter ŌåÆ verify.
2. Migrate the research adapter ŌåÆ verify.
3. Normalize + migrate works ŌåÆ verify.

**Guardrails every stage:**
- All 9 E2E specs stay green (`npx playwright test`) ŌĆö garden/research mount specs invariant; works spec updates in stage 3.
- Manual dev-server parity pass per graph: node color/shape/glyph; drag + Shift-pin; zoom/pan; reset view/positions; each filter dimension; panel open/close + focus return; position cache surviving reload; section-specific click (gardenŌåÆstack, researchŌåÆtheme/question URL, worksŌåÆ`node.url`); reduced-motion path.
- **CSS invariant:** core emits byte-identical class names per section so ┬¦28/┬¦31/┬¦36 keep matching. The *only* intended CSS changes are the works normalization.

## Definition of done

`graph-core.js` + 3 adapters replace the 3 monolith files; no behavioral change; 2,280 ŌåÆ ~900ŌĆō1,100 LOC; all 9 E2E green (works spec updated to div-wrapper); manual parity confirmed on all three pages + panels; `graph-core.js` at `assets/js/` depth. Then R5.3 (AMS-block + graph-panel consolidation ŌĆö subsumes R3.2's deferred structural piece) opens.

## Non-goals (YAGNI)

- No new graph features, no visual redesign, no d3 version change.
- No `research`/`works` N-hop or stack-coordination (garden-only by design).
- No change to the `graph-data.html` Hugo partials or the `graph-legend.html` / `graph-launcher-bar.html` already-shared partials.
