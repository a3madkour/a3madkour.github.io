# R5.2 — graph-core extraction Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Collapse three near-duplicate d3 graph runtimes (`garden-graph.js`, `research-graph.js`, `works-graph.js` — 2,280 LOC, ~70–75% shared) into one shared `assets/js/graph-core.js` + three thin adapters, with **zero behavioral change**.

**Architecture:** `graph-core.js` exports `createGraph(adapter)` → controller `{ rebuild, resetView, resetPositions, openPanel, closePanel }`. Core owns everything identical across the three today (cache, drag, zoom, settle loop, SVG scaffold, panel open/close on `inert`, panel-resize, chip primitives), parameterized only by `classPrefix`. Each adapter supplies the divergent surface (`parseData`/`applyFilters`/`nodeRadius`/`renderNode`/`forceParams`/`onNodeClick` + optional `onSvgCreate`/`onRenderComplete`/`onOpenPanel` hooks). Works is normalized to the garden/research conventions (JS toolbar / `inert` panel / `<div>` canvas). Staged migration garden→research→works.

**Tech Stack:** hand-rolled ES modules bundled by Hugo `js.Build` (esbuild), vendored d3 (force/zoom/drag/selection) dynamically imported, Playwright E2E (R5.1 harness).

## Global Constraints

- **`assets/js/graph-core.js` MUST stay at `assets/js/` depth** — the runtime `await import('./vendor/d3-*.min.js')` path is relative to the compiled bundle at `js/`. A subdirectory breaks the d3 import.
- **Zero behavioral change.** This is a pure refactor: node color/shape/glyph, drag+Shift-pin, zoom/pan, reset view/positions, filter chips, panel open/close+focus return, position-cache-survives-reload, and the section-specific click must all behave exactly as before.
- **CSS class names emitted by core must stay byte-identical per section** so existing `main.css` §28/§31/§36 rules keep matching. Core parameterizes via `classPrefix` ("garden"|"research"|"works") → `${prefix}-graph-node`, `${prefix}-graph-edge`, `${prefix}-graph-nodes`, `${prefix}-graph-edges`, and the zoom filter `.${prefix}-graph-node`. **The only intended CSS changes are the works normalization.**
- **The d3-selection `interrupt` stub lives once in core** (guarded `if (typeof selProto.interrupt !== 'function')`), shared across adapters via the browser module cache.
- **Garden-only features (stack-coordination, N-hop local mode, dynamic JS legend) never enter core** — they live in garden's adapter via the optional hooks.
- **Verification is E2E + manual parity.** After each stage: `npx playwright test` must stay green (9 specs), and a human/controller runs the parity checklist on the dev server (`hugo server`). There are no unit tests for graph JS.
- **LOC target:** 2,280 → ~900–1,100.
- Reference: design `docs/superpowers/specs/2026-07-04-r5.2-graph-core-design.md`; recon in the brief `docs/superpowers/specs/2026-07-04-r5.2-graph-core-brief.md`.

## The adapter contract (produced by Task 1, consumed by all)

Every adapter is a plain object with this exact shape. `createGraph(adapter)` reads these:

```js
// Node: { slug, title, url?, degree?, tag?, tags?, kind?, stage?, status?,
//         medium?, featured?, themeColorIdx?, x?, y? }
// Edge: { source, target, crossTopic?, kind?, via?, weight?, shared? }
const adapter = {
  classPrefix: 'garden',                 // string — drives all emitted class names
  pageSelector: '.garden-graph-page .garden-graph-canvas', // standalone-page canvas
  // panel canvas is always '.graph-panel-canvas' (shared) — core knows this.
  parseData() { /* → { nodes, edges } */ },
  applyFilters(filters) { /* (filters) → { nodes, edges } */ },
  filterCacheKey(filters) { /* (filters) → string, folded into the cache key */ },
  nodeRadius(node) { /* → number */ },
  renderNode(g, node) { /* draw shape/color/glyph into the core-created <g> */ },
  forceParams: {
    linkDistance: 60, linkStrength: 0.6, chargeStrength: -180,
    collideRadius(node) { /* → number */ },
  },
  onNodeClick(node, el) { /* navigation */ },
  buildToolbar(host, controller, filters) { /* build filter+action chips */ },
  // optional lifecycle hooks (core calls if present, else no-op):
  onSvgCreate(svg) {},         // e.g. works injects <defs> gradient
  onRenderComplete(controller) {}, // e.g. garden syncs .in-stack markers
  onOpenPanel(root) {},        // e.g. garden fills the dynamic legend
};
```

`createGraph(adapter)` returns:

```js
{ rebuild(), resetView(), resetPositions(), openPanel({ animate }), closePanel() }
```

Core owns a `state` container: `{ data, simulation, svg, viewTransform, pinnedSlugs, panel, filters }`. Chip clicks mutate `state.filters` (semantics defined by the adapter's `buildToolbar`) and call `controller.rebuild()`. Core passes `state.filters` to `adapter.applyFilters` and `adapter.filterCacheKey`.
